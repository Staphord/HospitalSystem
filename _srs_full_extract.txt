1. Introduction 

1.1 Purpose 

This  Software  Requirements  Specification  (SRS)  v1.0  defines  the  functional  and  non-functional 
requirements for the Hospital Patient Flow Management System deployed as a Multi-Tenant Software as 
a Service (SaaS)  platform.  The  system  serves multiple hospitals  simultaneously,  where  each  hospital 
operates as an independent tenant with complete data isolation from all other hospitals. 

1.2 Multi-Tenant SaaS Model 

The system operates under the following model: 

Layer 

Description 

System Owner 

Hospital Tenant 

Subscription 

Data Isolation 

The company that builds, hosts, and maintains the Hospital Patient Flow System. 
Manages all hospital tenants through a Super Admin portal. 

Each hospital that subscribes to use the system. A hospital is a self-contained tenant with 
its own data, staff accounts, configurations, and subscription. 

Each hospital pays a recurring subscription fee (monthly or annual) to access the system. 
Access is suspended if the subscription lapses. 

Each hospital has its own dedicated database. No data from Hospital A is ever accessible 
to Hospital B, their staff, or their administrators. 

1.3 Deployment Architecture Overview 

The recommended architecture uses a separate database per hospital tenant: 

•  One central application server hosts the system for all tenants. 
•  Each hospital is provisioned its own isolated database upon subscription activation. 
•  A tenant identifier routes each login session to the correct hospital database. 
•  The Super Admin portal connects to a separate master database tracking tenant accounts and 

subscriptions. 

•  No queries cross tenant database boundaries under any circumstance.

Tenant Context Rule (Applies to All Modules) 

TENANT ISOLATION RULE 
Every  data  query,  record  creation,  update,  deletion,  and  report  across  all  modules  must  be  automatically 
scoped to the logged-in user's hospital tenant. It is architecturally impossible for a user of Hospital A to read, 
write, or interact with any data belonging to Hospital B. This rule is enforced at the database connection level, 
not solely at the application code level.

have read-only access. They shall not be able to create, modify, or 
delete any patient or staff data. All impersonation sessions shall 
display a visible banner indicating the session is being monitored. 

Database credentials for each tenant's database shall be unique per 
tenant and stored in an encrypted secrets manager. No single 
credential shall grant access to more than one tenant database. 

The system shall undergo an independent penetration test at least 
annually, with specific focus on cross-tenant data leakage scenarios. 
Results and remediation must be documented.
